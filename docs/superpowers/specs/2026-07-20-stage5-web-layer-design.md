# Stage 5 — static web layer — design

## Purpose

Stages 1-4 produce `data/processed/distance_evaluation_results.gpkg`: one row
per evaluable candidate address (9,838 of them) with a strict/lenient
pass/fail verdict, margins, binding classifications, and (as of the
nearest-competitor-identity addition) the specific competitor behind each
distance figure. That file is keyed by `id_porpk` and carries only an
internal ID plus geometry — no human-readable address, and no context on
what currently occupies the address. Stage 5 assembles the single static
file the eventual map website (Stage 6) will load directly: one GeoJSON
feature per evaluable candidate, in web-ready EPSG:4326, with a readable
address label and today's occupancy context joined back in.

## Prerequisite

`distance_evaluation_results.gpkg` on disk predates the nearest-competitor-
identity addition (2026-07-19) and must be regenerated by re-running
`scripts/08_compute_distances.py` before Stage 5 can consume the 28
competitor-identity columns. This is a one-time re-run, not a Stage 5
responsibility, but Stage 5's script should fail loudly (not silently
proceed with stale/missing columns) if those columns aren't present.

## Scope

Only the 9,838 rows already in `distance_evaluation_results.gpkg` — no
filtering needed, that file is already scoped to "inside a ZPAE zone AND
matched to a classified street." Addresses outside ZPAE scope entirely are
out of scope for this tool and not included.

Strict is the headline verdict (marker color / primary popup text in the
eventual map); lenient and `interpretations_disagree` ride along as detail
fields under the same names Stage 4 already uses.

## Architecture

New script: `scripts/09_build_web_layer.py`. New module: `src/web_layer.py`
(pure, testable join/formatting functions), following the existing
`src/` (logic) vs `scripts/` (orchestration) split used by every prior
stage.

### Data flow

1. Load `distance_evaluation_results.gpkg` (EPSG:25830, 9,838 rows, the base).
2. Load `rt_portalpk_p_zpae_clip.gpkg` (Stage 2's raw address points —
   still on disk, has `tvia`, `nombre`, `numero`; confirmed 1:1 on
   `id_porpk`, no duplicates). Join in an `address` label.
3. Load `candidate_addresses_zpae_tagged.gpkg` (Stage 2/7 output). Join in
   `has_commercial_local`, `current_activity_summary`,
   `is_existing_hosteleria_class`.
4. Reproject the candidate's own geometry to EPSG:4326.
5. Reproject each of the 4 nearest-competitor lookups' `x`/`y` pairs
   (EPSG:25830) to lon/lat — these are plain float columns, not GeoDataFrame
   geometry, so they need their own `pyproj.Transformer` pass, separate
   from step 4.
6. Parse `current_activity_summary` back from its GPKG-round-tripped JSON
   *string* into a real Python list-of-dicts, so the final GeoJSON has
   proper nested JSON in that property rather than a double-encoded string.
7. Write `data/processed/zpae_viability_map.geojson`.

### `src/web_layer.py`

```python
def build_address_label(tvia: str, nombre: str, numero: str) -> str:
    """e.g. ("CALLE", "ARGANZUELA", "2") -> "Calle Arganzuela, 2"."""

def join_address_labels(
    results_gdf: gpd.GeoDataFrame, portal_gdf: gpd.GeoDataFrame
) -> gpd.GeoDataFrame:
    """Left-join results_gdf to portal_gdf on id_porpk, adding an
    'address' column built via build_address_label. portal_gdf must have
    tvia, nombre, numero, id_porpk."""

def join_occupancy_context(
    results_gdf: gpd.GeoDataFrame, tagged_gdf: gpd.GeoDataFrame
) -> gpd.GeoDataFrame:
    """Left-join results_gdf to tagged_gdf on id_porpk, adding
    has_commercial_local, current_activity_summary (parsed from its
    GPKG JSON-string form into a real list-of-dicts), and
    is_existing_hosteleria_class."""

def reproject_competitor_locations(
    gdf: gpd.GeoDataFrame, x_col: str, y_col: str, source_crs: str
) -> tuple[list[float], list[float]]:
    """Reproject a competitor-location x/y column pair (EPSG:25830) to
    lon/lat, returning two lists to assign back as new columns. None
    values pass through as None."""
```

### `scripts/09_build_web_layer.py`

Orchestrates: load the three inputs, assert the 28 competitor-identity
columns are present (fail loudly per Prerequisite above), call
`join_address_labels` and `join_occupancy_context`, reproject the main
geometry to EPSG:4326, call `reproject_competitor_locations` for each of
the 4 `{strict,lenient}_nearest_{binding,overall}_{x,y}` pairs (writing
results to new `..._lon`/`..._lat` columns, keeping the original `x`/`y`
EPSG:25830 columns too, so no information is lost), parse
`current_activity_summary`, and write the final GeoJSON.

## Output

`data/processed/zpae_viability_map.geojson` — one Point feature per
evaluable candidate, EPSG:4326. Properties: everything already in
`distance_evaluation_results.gpkg` (zone, classification, pass/fail,
margins, binding classifications, `prohibited_outright`,
`interpretations_disagree`, all 28 competitor-identity fields plus their
new `_lon`/`_lat` companions) plus `address`, `has_commercial_local`,
`current_activity_summary` (real nested JSON), `is_existing_hosteleria_class`.

## Testing

Unit tests in `tests/test_web_layer.py`:
- `build_address_label`: normal case, and a case with a non-numeric/blank
  `numero` (some portal points have `extension: "No aplicable"` — confirm
  the label degrades gracefully rather than producing a malformed string).
- `join_address_labels`: correct label attached, left-join semantics
  (a `results_gdf` row with no matching `id_porpk` in `portal_gdf` keeps its
  other columns with a null address rather than being dropped).
- `join_occupancy_context`: correct fields attached; `current_activity_summary`
  round-trips from its JSON-string GPKG form into a real Python list.
- `reproject_competitor_locations`: a known EPSG:25830 coordinate reprojects
  to the expected EPSG:4326 lon/lat within a small tolerance; `None` values
  pass through as `None` rather than raising.
